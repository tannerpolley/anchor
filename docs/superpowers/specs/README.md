# Specs

Use this root for loose product, design, PRD, and architecture specs.

Specs may reference milestones, categories, or future issue candidates, but they do not need GitHub issue metadata or execution fields. When a spec becomes implementation-facing, create a plan under `docs/superpowers/plans`.

Recommended frontmatter:

```yaml
title: Short spec title
milestone: M1 - Issue Workflow Hardening
status: draft
owner: codex
```
